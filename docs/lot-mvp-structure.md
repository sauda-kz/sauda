# Структура lot и lot_match (MVP — SAUDA-006A)

> Источник правды в коде: миграции `V2__init.sql` + `V3__lot_mvp_structure.sql`, JPA `Lot`, `LotMatch`.  
> ER-диаграмма: [er-diagram.md](er-diagram.md). RBAC: [auth-rbac.md](auth-rbac.md).

## Назначение

В MVP Sauda закупочные лоты добавляются **вручную или полувручную** администратором платформы по структурированным полям. Система **не** читает PDF/DOCX и **не** извлекает требования из документов автоматически.

Матчинг (`lot_match`) связывает лот с предложением дистрибьютора (`offer`) и показывает дистрибьютору раздел **«Подходящие лоты»**.

---

## Сущность `lot`

Закупочный лот — внешний запрос заказчика: что купить, в каком количестве, на какую сумму, куда поставить и какие требования выполнить.

| `procurement_method` | нет | ручное | Способ закупки |
| `lot_type` | да | ручное | товар / работа / услуга / комплексный |
| `quantity` | да | ручное | Требуемое количество |
| `unit` | да | ручное | Единица измерения |

### Статусы `lot` (`lot_status`)

| Статус | Описание |
|--------|----------|
| `active` | Лот активен, доступен для матчинга |
| `archived` | Архивирован |

## Сущность `lot_match`

Связь лота с предложением дистрибьютора. Колонка в БД: `match_status`; в API DTO — поле `status`.

| Поле | Обязательное (MVP) | Заполнение | Описание |
|------|-------------------|------------|----------|
| `id` | — | авто | UUID |
| `lot_id` | да | ручное / логика | Ссылка на лот |
| `offer_id` | да | ручное / логика | Ссылка на предложение |
| `distributor_id` | да | авто | Должен совпадать с `offer.distributor_id` |
| `match_status` | да | ручное / логика | Статус матча (см. ниже) |
| `match_reason` | да | ручное (admin) | Почему предложение подходит |
| `required_quantity` | да | авто | Из `lot.quantity` |
| `available_quantity` | да | авто | Из `offer.stock_qty` (0 если null) |
| `needs_manual_review` | да | ручное / default `true` | Нужна ручная проверка |
| `admin_comment` | да | ручное (default `''`) | Комментарий админа |
| `distributor_comment` | нет | ручное (дистрибьютор) | Комментарий дистрибьютора |
| `confidence_score` | нет | ручное / условное | Уровень уверенности (0–1) |
| `matched_requirements` | нет | ручное | JSONB-массив совпавших требований |
| `missing_requirements` | нет | ручное | JSONB-массив несовпавших / непроверенных |
| `risk_flags` | нет | ручное | JSONB-массив рисков |
| `quantity_check` | — | логика | `ok` / `fail` / `unknown` |
| `stock_check` | — | логика | `ok` / `fail` / `unknown` |
| `price_check` | — | логика | `ok` / `fail` / `unknown` |
| `estimated_unit_price` | нет | авто | Из `offer.price` при наличии |
| `estimated_total_price` | нет | авто | unit_price × required_quantity |
| `budget_amount` | нет | авто | Снимок `lot.budget_amount` |
| `estimated_margin` | нет | авто | budget − estimated_total (если можно) |
| `created_at` | да | авто | Дата создания |
| `updated_at` | да | авто | Дата обновления |

Уникальность: `(lot_id, offer_id)`.

### Статусы `lot_match` (`lot_match_status`)

| Статус | Описание |
|--------|----------|
| `suggested` | Совпадение предложено системой или админом |
| `matched` | Совпадение подтверждено |
| `needs_review` | Требуется ручная проверка |
| `not_matched` | Предложение не подходит |
| `dismissed` | Дистрибьютор скрыл / отклонил |
| `interested` | Дистрибьютор заинтересовался |
| `mismatch_reported` | Дистрибьютор сообщил о неверном совпадении |

### Действия дистрибьютора

| UI-действие | `match_status` |
|-------------|----------------|
| Интересно | `interested` |
| Скрыть | `dismissed` |
| Нужна проверка | `needs_review` |
| Не подходит | `not_matched` |
| Сообщить о неверном совпадении | `mismatch_reported` |

## Что видит дистрибьютор в карточке лота

| Блок UI | Источник данных |
|---------|-----------------|
| Название, заказчик, сумма, сроки, место, категория, количество | `lot` |
| Подходящий товар | `offer.raw_name`, `offer.brand`, `offer.model_mpn` |
| Остаток, цена | `lot_match.available_quantity`, `estimated_*` / `offer.price` |
| Примерная маржа, уверенность | `lot_match.estimated_margin`, `confidence_score` |
| Причины совпадения | `match_reason`, `matched_requirements` |
| Риски и непроверенное | `risk_flags`, `missing_requirements`, `needs_manual_review` |
| Документы | `lot.required_documents` |
| Ссылка на закупку | `lot.source_url` |
| Действия | смена `match_status` |

Контракт API (`DistributorLotMatchCardResponse`) — реализован в SAUDA-006B.

| Метод | Путь | Назначение |
|-------|------|------------|
| `GET` | `/api/v1/lots` | Список лотов (platform) |
| `GET` | `/api/v1/lots/{id}` | Лот по ID |
| `POST` | `/api/v1/lots` | Создание лота |

## Логика MVP

1. Админ Sauda вручную создаёт `lot` и заполняет структурированные поля.
2. В системе есть предложения дистрибьютора (`offer`).
3. Админ или базовая логика создаёт `lot_match`.
4. Дистрибьютор видит лот в «Подходящие лоты» (фильтр по `distributor_id`).
5. Дистрибьютор меняет статус: interested / dismissed / needs_review / not_matched.
